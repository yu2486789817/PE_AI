3. 项目描述 
3.1 背景 
本项目旨在规划与开发一个面向高校体育教学的"AI 智慧体育课堂平台"。当前高
校体育教学普遍面临诸多挑战：师生课后互动严重不足，学生在课堂外难以获得专业
的训练指导；学生课外训练缺乏有效监督与科学评估机制；教学过程数据采集困难，
缺乏量化评估依据。特别是在后疫情时代，线上线下混合式教学已成为教育发展的重
要趋势，体育教学作为实践性极强的学科，亟需探索适合自身特点的数字化教学模式。  
3.2 问题/机会 
问题： 
• 传统的大学体育教学在课外缺乏个性化的指导与反馈 


===== PAGE 3 =====
• 面对大量的学生体育动作视频作业，教师手动批改耗时费力 
• 学生的错误动作得不到及时纠正，容易形成错误的肌肉记忆 
• 教师难以掌握学生的真实训练投入和进步情况 
机会： 
通过利用 AI 视频分析技术（如 YOLO）来自动评估学生练习姿势、实时纠正动作
并生成智能反馈，是一个极大的机会，这将显著提升教学效率和学生的学习效果。 
3.3 项目整体概述 
开发一个集成了 AI 自动视频分析、课程管理和作业追踪功能的智能体育教学平台。
平台采用 B/S 架构，基于云计算技术，主要包含： 
• Web 前端界面（学生端和教师端） 
• 移动端应用 
• Spring Boot 核心业务服务层 
• Python AI 服务层 
• MySQL 数据库 
3.4 预期影响 
项目完成后，将有力推动高校体育教学的标准化、个性化与数据化发展。对学生
而言，将提供个性化的训练陪伴与即时指导，提升训练的有效性和成就感；对教师而
言，通过直观的数据看板全面掌握班级学情，使教学决策建立在客观数据基础之上，
最终实现体育教学质量与学生体质健康的双提升。

5. 项目范围 
5.1 项目内范围 (In Scope) 
学生端功能： 
• 用户账户管理（注册、登录、个人信息维护） 
• 课程学习模块（通过课程码加入课程、观看教学视频、学习进度记录） 
• 训练作业模块（上传训练视频、查看 AI 分析结果、历史记录查询） 
• 智能交互模块（AI 对话功能） 
教师端功能： 
• 课程管理模块（创建、编辑、发布、归档课程） 
• 学生管理模块（学生信息导入导出、分班管理、学习进度监控） 
• 作业管理模块（发布、批阅、统计分析） 
• 教学分析模块（学情分析、数据可视化） 
技术基础设施： 
• Spring Boot 后端服务框架 
• Python AI 服务（YOLOv8 动作识别、智能对话服务） 
• MySQL 数据库 
• Docker 容器化部署 
• 前端 Web 页面 
• 基于 Uni-app 开发的跨平台移动端应用

• 与学校现有系统的深度集成（如统一身份认证）

10.2.1 风险等级定义

10.2.2 风险登记册 
风险登记册是风险管理的核心工具，记录了项目识别的所有重要风险及其应对策
略。本项目共识别出 10 项主要风险，涵盖技术风险、资源风险、管理风险和外部风险
四个类别。 
技术风险分析： 
技术方面最大的风险是动作识别的准确率问题。体育动作比较复杂，光线、角度、
服装等因素都可能影响识别效果。如果识别结果不够准确，给出的纠正建议就会不可
靠，影响用户体验。我们的应对策略是先聚焦几个基础动作，做好做精，而不是贪多
求全。初期聚焦基础动作（如深蹲、引体向上）的识别，使用公开运动数据集预训练
YOLOv8 模型，逐步引入真实场景数据优化模型鲁棒性。 
另一个技术风险是系统性能问题。视频处理比较耗费计算资源，如果多个学生同
时上传视频，系统可能会响应缓慢。我们计划采用任务队列的方式，让视频分析在后
台进行，不影响学生的其他操作。采用任务队列异步处理视频分析，优化资源分配，
引入 Redis 缓存，实现负载均衡。 
Spring Boot 与 Python 服务的集成也是一个潜在的技术风险点。两种技术栈的集
成需要定义清晰的接口规范，并进行充分的测试。我们计划在项目初期进行POC（概
念验证）测试，验证技术方案的可行性。 
资源风险分析： 
GPU 计算资源不足是主要的资源风险。视频处理对 GPU 资源需求较高，如果并
发用户数量超过预期，可能导致服务器负载过高。我们计划通过优化模型推理效率、
实现任务队列调度、限制视频分辨率和时长等方式来缓解这一风险。 
管理风险分析： 
团队成员时间冲突是主要的管理风险。作为学生团队，成员可能面临其他课程任
务、考试、实习等情况，影响项目投入时间。我们计划通过合理分配任务、建立备份
机制、提前识别风险、保持文档完善等方式来应对这一风险。 
项目进度延期也是一个需要关注的风险。我们计划设置合理的缓冲时间，每周进
行进度检查，及时调整计划。如果出现进度延期，将考虑削减非核心功能，延长项目
周期，或增加投入。 
外部风险分析： 
用户接受度是一个需要考虑的外部风险。老师和学生可能不习惯这种新的教学方
式，或者觉得操作太复杂。我们需要把界面做得尽量简单易用，同时提供充分的使用
指导。我们计划进行早期用户测试，根据用户反馈迭代优化界面。 
数据隐私合规问题也是一个重要的外部风险。我们的平台会收集学生的训练视频
和个人信息，这涉及到数据隐私保护问题。我们需要确保符合学校的相关规定，也要

10.3 限制因素 
限制因素是项目实施过程中必须遵守的约束条件，这些因素通常不可改变或难以
改变，项目团队需要在限制条件下寻求最优解决方案。 
10.3.1 时间限制 
作为高校课程项目，本项目必须在规定的学期时间内完成。时间限制是项目面临
的最主要约束之一，直接影响项目的范围界定和功能优先级排序。我们需要在有限的
时间内合理规划里程碑，设置优先级，确保核心功能优先完成。 
团队成员还有其他课程任务和考试，这也会影响人员投入时间。我们需要灵活安
排工作时间，提前协调考试周安排，确保项目进度不受影响。 
迭代周期固定为每两周一个迭代，这要求我们严格控制每个迭代的工作量，避免
过度承诺导致迭代目标无法完成。

11. 项目行政管理 
11.1 沟通计划 
本沟通计划用于明确项目全过程的信息传递方式、频率、责任人、接收对象与沟

利推进。 
1. 沟通目标 
• 确保团队成员清晰了解项目目标、进度、分工与风险。 
• 保证指导老师、项目相关方及时掌握项目状态。 
• 及时暴露问题、协调资源、推动决策。 
• 统一项目信息口径，避免沟通混乱与范围蔓延。 
2.沟通类型与频率 
• 每日站会（简短同步）形式：线上文字 / 语音内容：昨日完成、今日计划、遇到障
碍频率：工作日每日 1 次，5 分钟以内责任人：项目经理 
• 周进度会议时间：每周一晚上时长： 30–40 分钟内容：进度对比、任务完成情况、

• 月度总结会议时间：每月最后一周内容：阶段目标达成情况、成果展示、计划调

• 专项问题会议触发条件：出现重大技术风险、进度延误、范围变更形式：随时召
开，快速决策输出：问题解决方案与行动项 
3.沟通渠道与工具 
• 即时沟通：微信群 / QQ 群（日常通知、快速问题、文件共享）

量、资源的影响，并在 1 天内完成初步评估。 


===== PAGE 31 =====
第三步：决策。根据变更类型采取不同的决策方式：小型变更由提出人与项目经理协
商决定；中型变更在团队会议上讨论，多数同意后通过；重大变更需团队讨论后向指
导教师汇报，获得批准后方可执行。 
第四步：记录与执行。无论变更是否被接受，都需要记录在变更日志中。被接受的变
更要及时更新任务分配和项目计划；被拒绝的变更要说明原因并记录。

11.4 人力资源计划 
11.4.1 团队角色与职责 
根据团队成员的专业背景和技术特长，任务分配如下： 
项目经理负责项目整体统筹，包括制定计划、跟踪进度、组织会议、风险控制和文档
管理。 
技术负责人负责 AI 模型开发和技术决策，包括 YOLOv8 模型训练优化、动作识别算法
实现、技术方案选型。 
后端开发工程师负责后端系统开发和数据库设计，包括 API 接口开发、数据库维护、
服务器部署和系统集成。 
前端开发与设计工程师负责前端界面开发和交互设计，包括教师端和学生端界面开发、
响应式适配、界面优化。 
11.4.2 团队协作机制 
日常沟通方面，团队通过每日站会同步进度，每人简要说明昨天完成了什么、今
天计划做什么、遇到什么困难；每周召开一次周例会，总结本周工作、规划下周任务、
讨论重要问题；技术问题按需随时讨论。 
代码审查方面，为确保代码质量，所有代码提交后需由技术负责人或指定成员进
行审查，审查内容包括代码规范性、逻辑正确性、性能影响和安全隐患。审查通过后
代码才能合并到主分支。 
知识传递方面，团队定期进行技术分享，每人轮流分享自己负责模块的技术点或
近期学到的新知识。同时所有技术文档、设计文档和会议记录统一存放在团队共享空


===== PAGE 33 =====
间，方便查阅和传承。 
11.4.3 任务协作原则 
任务分配遵循“专业对口、能力匹配”的原则，尽量让每个人负责自己擅长的工作。
遇到困难时及时求助，不硬扛。重要任务采用结对方式，一人开发一人审查，减少错
误。同时团队成员之间互相备份关键知识，避免因某人生病或忙碌导致工作停滞。 
11.5 项目收尾计划 
11.5.1 收尾目标 
项目收尾阶段的目标是确保所有工作成果完整交付，项目经验得到沉淀，为后续
项目或课程答辩做好准备。 
11.5.2 最终交付文档 
项目完成时，需要整理并提交以下文档： 
• 项目章程：最终版本的项目章程 
• 需求规格说明书：完整的功能需求和非功能需求说明 
• 系统设计文档：架构设计、数据库设计、接口设计文档 
• 用户操作手册：教师端和学生端的操作指南 
• 测试报告：功能测试、性能测试的结果记录 
• 项目总结报告：项目整体回顾、经验教训总结 
11.5.3 代码与文档归档 
所有代码需提交到代码仓库（如 GitHub 或 Gitee） ，并打上最终版本标签。代码仓
库中需包含清晰的 README 说明，介绍项目结构、部署方式和运行方法。 
所有文档需整理到团队共享空间（如飞书文档、腾讯文档或网盘） ，按文档类型分
类存放，确保完整、有序、可查阅。 
11.5.4 项目经验教训总结 
项目结束后，团队召开复盘会议，讨论以下问题： 
• 项目目标是否达成？哪些做得好？哪些做得不好？ 
• 过程中遇到的主要问题是什么？如何解决的？ 


===== PAGE 34 =====
• 如果重新做一次，哪些地方可以改进？ 
• 对后续类似项目有什么建议？ 
复盘会议记录整理成《项目经验教训总结》 ，作为团队的知识资产。 
11.5.5 系统演示和答辩准备 
演示准备方面，提前准备好演示环境（本地或服务器） ，确保系统稳定可用。设计
完整的演示流程，覆盖项目的核心功能，提前演练 1-2 次，确保演示流畅不卡顿。同
时准备好备选方案，万一现场出现技术问题，有应急措施（如录屏备用） 。 
答辩准备方面，制作答辩 PPT，内容包括：项目背景与目标、核心功能与创新点、
技术方案与实现、项目成果展示、总结与展望。团队成员提前讨论可能被问到的问题，
统一回答口径，确保答辩时配合默契。
